## Creating a Pull Request

<Steps>

1. **Ensure your branch is up to date**

   ```bash
   git fetch origin
   git rebase origin/main
   ```

2. **Run all checks locally**

   ```bash
   bun run lint
   bun run type-check
   bun run test
   ```

3. **Push your branch**

   ```bash
   git push origin feature/your-feature
   ```

4. **Open a Pull Request**

   Go to GitHub and click "New Pull Request".

5. **Fill in the PR template**

   See template below.

</Steps>

## PR Template

```markdown
## Description
Brief description of changes.

## Changes
- Change 1
- Change 2
- Change 3

## Related Issues
Closes #123

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation

## Testing
- [ ] Unit tests pass
- [ ] Integration tests pass
- [ ] E2E tests pass (if applicable)
- [ ] Manually tested

## Screenshots
[Add if UI changes]

## Checklist
- [ ] Code follows style guidelines
- [ ] Self-reviewed my code
- [ ] Added/updated tests
- [ ] Updated documentation
- [ ] No new warnings
```

## PR Requirements

### Must Have

- Clear description of changes
- Passing CI checks
- Tests for new functionality
- No merge conflicts

### Should Have

- Related issue linked
- Screenshots for UI changes
- Documentation updates

## Review Process

### For Authors

1. Keep PRs focused and small (under 500 lines when possible)
2. Respond to feedback promptly
3. Request re-review after changes
4. Squash commits when merging

### For Reviewers

1. Review within 24-48 hours
2. Be constructive and respectful
3. Ask questions to understand intent
4. Approve when ready, don't block on minor issues

**Review Checklist:**

- [ ] Code is readable and maintainable
- [ ] Tests are adequate
- [ ] No obvious bugs or security issues
- [ ] Changes are backward compatible

## Merging

Once approved:

1. Squash and merge (default)
2. Write a clear merge commit message
3. Delete the feature branch
4. Verify deployment (if applicable)
